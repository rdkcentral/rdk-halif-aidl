    Vendor Test Suite (VTS)

    1. Introduction

    The Vendor Test Suite (VTS) is a comprehensive validation framework designed to verify the correctness, robustness, and compliance of vendor-layer implementations, including HAL interfaces and platform services.

    VTS ensures that vendor-specific implementations meet expected specifications and behave reliably before integration with upper layers, enabling early issue detection and improved system stability.

    VTS is not just a test suite — it acts as a vendor validation platform supporting development, integration, and acceptance workflows.

    2. Why VTS?

    Modern platforms involve multiple layers (HAL, vendor, middleware, applications), often developed independently. This introduces challenges such as:

    • Late detection of integration issues
    • Inconsistent validation across vendors
    • Difficulty in isolating defects
    • Limited early-stage automation

    VTS addresses these challenges by:

    • Enabling early-stage (shift-left) validation
    • Standardizing testing across vendors and platforms
    • Providing repeatable and automated test execution
    • Supporting black-box testing that validates purely against the specifications
    • Enabling Easy Integration and Certificate validation

    3. Key Capabilities

    • Platform-agnostic architecture
    • Modular and extensible test design
    • YAML-driven test definitions
    • Integration with automation frameworks (RAFT)
    • Support for rack-based and desk-based testing
    • Scalable from unit testing to system validation
    • Future-ready (AI-assisted test generation, DevOps integration)

    4. Who Should Use VTS?

    HAL Developers

    • Validate interface compliance
    • Perform parameter and boundary testing

    Vendors (SoC Providers)

    • Validate vendor layer implementations
    • Ensure platform compatibility and Interface Compliance

    Integrators

    • Validate multi-module interactions
    • Detect integration issues early

    QA / Product Teams

    • Perform layer-level validation
    • Perform Multi-layer (system) tests to validate the Vendor Layer functionalities.

    Where VTS Fits in Development Lifecycle

    VTS supports validation across the complete vendor development lifecycle:

    • Development Stage
        Enables developers to validate HAL interfaces and module behavior early using L1 and L2 tests.

    • Module Integration Stage
        Detects issues when individual modules are combined, ensuring correct inter-module interactions.

    • Layer Integration Stage
        Validates that the vendor layer complies with specifications before integration with middleware and application layers.

    • System Integration Stage
        Uses L3 tests with external stimulus to validate real-world interactions and multi-module workflows.

    • QA / Acceptance Stage
        Supports system-level validation (L4), including performance, stability, and pre-compliance testing for release readiness.

    5. VTS Test Levels

    L1 – Component Function Test (API Validation)

    • Parameter validation (min/max/boundary)
    • Positive and negative testing
    • Error code validation
    • Ensures API compliance

    L2 – Component Unit Testing

    • Functional validation of individual modules
    • No external stimulus
    • Validates: State changes, Get/Set operations, Precondition checks

    L3 – Integration / Functional Testing

    • Multi-module interaction
    • External stimulus involved
    • Black-box testing approach
    • Controlled via RAFT framework

    Examples:

    • HDMI hotplug with analyzer
    • CEC communication via adapter
    • Audio Video Validation with Test-specific streams

    L4 – System Testing

    Comprehensive validation including:

    • Multi-layer interaction
    • Performance testing (CPU, memory, boot time)
    • Stability (stress testing)
    • Pre-compliance (power, behavior)

    Additional Test Categories

    • Smoke Tests
    • Performance Tests
    • Stress Tests
    • Power Management Tests

    For more information, refer to Test Levels

    8. Getting Started with VTS - Engineer Workflow

    This section outlines the recommended sequence for engineers to begin using VTS to validate a module or platform.

    Step 1: Understand the Scope

    • Identify the HAL module/s to be validated (e.g., HDMI CEC, Audio Decoder, etc.)
    • Review supported APIs and expected behaviors. Make sure the supported modules are validated before using them.
    • Identify dependencies (other modules, external devices) based on the Test procedure documents as defined in text

    Step 2: Set Up the Environment

    • Build VTS binaries
    • Ensure DUT (Device Under Test) is accessible
    • Verify connectivity (SSH / ADB / network access)

    Optional: - Set up rack infrastructure (HDMI analyzer, CEC adapter, etc.) - Configure RAFT if an external stimulus is required

    Step 3: Setting up the Configurations

    • Update the Device and Rack Configs correctly based on the environment
    • Update the Module hfp profiles based on the platform capabilities
    • Make sure that all the required software packages are installed

    Step 4: Understand what needs to be tested

    • Running Manually - Refer Manual Test runs
    • Driving from Host - Refer Auto Test runs
    • Single tests or Multiple tests that need to be run in an automatic environment are defined in the above test procedures.

    Step 5: Test Results and Reports

    • Manual Test reports can be viewed once the test is executed.
    • Automatic Test reports and logs are generated under the folder "logs//rack#/slot#/TestModuleName/".

    9. Best Practices

    • Suggested to start with L1 during development
    • Gradually enable higher test levels
    • Automate execution in CI/CD pipelines
    • Maintain clear and consistent YAML definitions
    • Use detailed logging for debugging
